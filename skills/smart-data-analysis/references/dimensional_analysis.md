# 数据洞察 · 场景：维度分析（Dimensional analysis）

> **读取时机（MUST）**：当且仅当本轮 **启用维度分析场景** 时，须在 **子场景识别已完成**、且已完成新版 smart-ask-data **第 7～9 步**（候选表检索 → 权限校验 → `dataview get` **收敛可查询的数据视图与字段**）之后、编写子问题清单之前，阅读本文至少 **「子问题拆分规则」** 与 **「输出模板（维度分析章节）」** 两节。**禁止**未读模板即列出子问题；未启用本场景时 **禁止**整本预读。与其它子场景（含 `attribution_analysis.md`）不必同时预读。

本场景强调：**多角度 / 多维度 / 拆解 / 结构 / 分布 / TopN** 等结构化洞察，**非**时间序列叙事主线的可优先用本模板；与 **数据解读**（趋势/异常/贡献章节）、**对比分析**（双组或多期对照）互补。

## 强约束提示（MUST：启用本子场景时强制执行）

当本轮被识别为「维度分析」子场景时，**对用户可见的输出必须严格按本文「输出模板（维度分析章节）」的章节顺序与组织方式呈现**，不得用其它场景（如 `data_interpretation.md` ）替代本模板。

- **结构强制**：必须依次输出且不得省略以下 4 节：  
  1)「从哪些角度看了数据」→ 2)「各角度的主要发现」→ 3)「建议关注的重点与后续动作」→ 4)「局限与下一步」  
- **表格强制**：在「各角度的主要发现」中，**每个有数据支持的角度必须给出 Markdown 表格**（不能只写结论段落）。  
- **表格优先解释（MUST）**：当结论涉及具体数字（数量、金额、占比、排名、差值等）时，**优先把数字放进表格**，文字只做 2～4 句解释；避免只写“很多/较高/集中”而不配表。  
- **排序强制**：表格行顺序必须与查询结果或 `ORDER BY` 一致，**不得**为了叙事重排。  
- **TopN 截断强制**：若任何角度结果为 TopN（显式 `LIMIT N`、或仅展示部分行），主文必须明确写出：  
  - 展示的是 TopN 还是全量；若为 TopN，说明「其余归入其他/未展示」  
  - 若写占比，必须写清分母含义，且仅在可验算前提下给出  
- **证据绑定强制**：本子场景的每个关键数字必须能回指到某条子问题的 `smart-ask-data` 第 11 步查询结果；**禁止**在未取数的角度上写结论（可以写“本轮未统计××，原因…”）。  
- **避免混写**：允许在同轮同时启用对比分析/数据解读，但维度分析章节本身仍须按本文模板输出；其它场景的内容应放在各自模板中，不得把维度分析模板“挤掉”。  
- **结论验证对齐（MUST）**：在进入最终输出前，必须对本章关键结论做可复核校验：  
  - **加总/占比可验算**：能从结果表中验算合计与占比；TopN 截断必须明确“仅展示前 N 条”  
  - **口径不漂移**：同一指标在不同角度引用时分母/过滤条件一致（或显式声明不同口径）  
  - **限行不外推**：若任何角度来自明细 `LIMIT` 样例，不得据此推断全量结构/占比；需要全量结构必须改为聚合子问题取数  

**对用户正文的语言（MUST）**

- **主文**：用业务读者熟悉的说法——如「从地区看」「从类型看」「排名靠前的是」「大部分集中在」；避免通篇「维度 D」「度量 M」「分组字段」等 **编排用语**（这些仅用于 **对内** 子问题设计）。
- **表格**：列名优先 **业务中文**（如「地区」「案件数」「罚没金额合计」）；若与库表字段名不同，在「附录：核对用」中说明对应关系。
- **证据**：TopN、占比、合计仍须可回到子问题结果；SQL 与子问题编号建议放在 **附录**，不插在每段分析中间。

## 子问题拆分规则（MUST）

**必须先识别**（写在「口径与证据来源」或对内草稿中，可对用户简述）：

- **分析维度（D）**：用于分组的类别字段或层次（如地区、类型、渠道、状态、时间粒度等）；须与新版 smart-ask-data 第 9 步 `dataview get` 返回的 `fields` 字段可对应。
- **度量（M）**：计数、金额、均值等；须可用 **一条可执行 SQL** 表达，并在新版 smart-ask-data **第 10～11 步**（生成 SQL → `dataview query` 执行）中完成取数（如 `COUNT(DISTINCT id)`、`SUM(amount)`）。

**规则**：

- **每个主要维度至少一条子问题**：例如「按地区案件数」「按违法类型案件数」「按月份趋势」各为一条；避免一条 SQL 强行堆叠过多维度导致不可读或方言不支持，除非单条 SQL 已能稳定表达且仍满足 `smart-ask-data` 规范。
- **每条子问题**输出须为 **一次** `smart-ask-data` 的完整交付（原样结果 + 最小口径；SQL 如存在仅内部留存不对用户展示）；该交付即 **该维度片段** 的洞察输入。
- **合成**：在「各维度的洞察结果」中按维度逐段引用对应子问题的结果；**不得**把未查询过的维度写进结论。

## 输出模板（维度分析章节；MUST：启用本子场景时强制按照这个模板来输出）

> 适用前提：至少一条子问题的结果集中存在可解释的 **维度字段** 与 **度量字段**（编排侧用语）。不满足则写「不适用」，并给出下一步取数建议。
> **标题括号注释禁止输出（MUST）**：每一章节标题后面括号内的内容仅为技术性说明，**对用户可见输出禁止包含括号及其中内容**；仅输出括号前的小标题文本。

### 从哪些角度看了数据（必选；对用户用业务标题）

- 用列表说明本轮 **从哪些业务角度** 做了统计（如「按处罚决定年份」「按违法类型」「按企业登记机关」）；每项一句 **能看出什么**。
- **对内对照**（可放在附录）：每个角度对应哪条子问题、用了哪些字段；若某角度未取到数，主文写「本轮未统计××，原因…」。

### 各角度的主要发现（必选，按角度分段）

对每个 **有数据支持的角度**，先写 **2～4 句业务概括**，再给表格。表格建议列名：

| 分类（或地区 / 类型等业务名） | 数量或金额等业务指标 | 名次 | 简要说明（可选） |
| --- | ---: | ---: | --- |

（排序须与查询结果 / `ORDER BY` 一致，不得改序。）

并给出：

- **头部情况**：哪几项排在前、具体数字多少（主文用「第一、第二…」；**子问题编号放附录**）。
- **集中程度**：若可验算，用业务话写「前 N 项合计约占…%」并写清分母含义；若结果为 TopN **截断**，主文须明确「仅展示前 N 条，其余归入其他」。
- **分布形态（可选）**：若输入支持，用一两句描述「是否一家独大 / 相对平均 / 长尾明显」等。

### 建议关注的重点与后续动作（必选）

3～7 条，区分：

- **可立刻推进**：基于当前统计的核查、督导、资源配置或业务跟进点（不写 SQL）。
- **需要再取数才能定**：用业务话写缺什么信息（如「若要知道是否季节性，需要按月再统计」）；技术落地方式可简短附在括号内。

### 局限与下一步（必选）

- 主文说明 **本轮回答回答不了什么**（业务表述）。
- **附录（建议）**：子问题编号 ↔ 角度 ↔ **取数逻辑摘要（业务话）** ↔ 结果表索引，便于复核（对用户输出禁止展示 SQL）。

## 与对比分析、数据解读的协作

- 若本轮 **同时** 启用 **对比分析**：维度章节中的「时间」类统计应避免与对比章节 **口径矛盾**；优先在对比章节写 **两期谁高谁低**，在维度章节写 **单期内** 结构（主文均用业务表述）。
- 若本轮 **同时** 启用 **数据解读**：数据解读第 5 节「关键贡献维度」可与本节 **引用同一子问题交付物**，**禁止**重复编造数字；用户可见输出可在数据解读中摘要、在维度章节给表，或仅在一处展开并另一处写「见上文 ×」。对用户最终输出 **不展示 SQL**；若需要复跑/审计，仅在内部留存 SQL，并在用户可见附录中用「取数逻辑 + 证据索引」做对照。
