# 数据洞察 · 场景：数据解读（Data interpretation）

> **读取时机（MUST）**：当且仅当本轮 **启用数据解读场景** 时，须在 **子场景识别已完成**、且已完成新版 smart-ask-data **第 7～9 步**（候选表检索 → 权限校验 → `dataview get` **收敛可查询的数据视图与字段**）之后、编写子问题清单之前，阅读本文至少 **「子问题拆解指引」** 与 **「输出模板」** 两节（下文以 `##` 标题为准）。**禁止**拖到全部第 11 步查询执行完成后再首次阅读模板；也 **禁止**在未启用本场景时整本预读本文。
>
> **与其它子场景**：不必与 `dimensional_analysis.md` / `contrastive_analysis.md` / `attribution_analysis.md` 同时预读；多场景启用时再分别打开对应文件。

本场景的定位：**在可复核的取数证据上，按固定章节输出解读结论**（趋势 / 异常与可能解释 / 关键贡献维度 / 建议）。与 **维度分析**、**对比分析** 场景共享同一套上游编排（**子场景识别** → 新版 smart-ask-data 第 7～9 步收敛可查询视图与字段 → **读本场景模板** → 子问题 → 第 10～11 步生成 SQL 并执行查询），差异仅在 **子问题映射目标** 与 **本章输出模板**。

**对用户正文的语言（MUST）**

- **主文**：面向业务读者——多用「增长/回落/集中在/占比高的是/建议优先关注」等说法；用 **业务含义** 解释指标（如「处罚金额合计」而非仅 `SUM(penam)`）；避免在趋势、异常、贡献、建议各节中堆砌表名、英文字段、命令细节、SQL。
- **证据不落空**：数字与判断仍须能在附录或「口径与证据来源」后半中 **逐条对应**到子问题结果（必要时用取数逻辑与证据索引定位）；主文与附录分工，而不是省略证据。对用户最终输出不展示 SQL，SQL 如需复跑仅内部留存。

## 触发与边界

**触发**：用户提到「解读数据」「分析结果」「给建议」「异常说明」「趋势」「贡献」等。若用户 **同时** 要求「归因/根因/MECE 证据链」，可与 [`attribution_analysis.md`](attribution_analysis.md) **同轮启用**；轻量解读不得替代归因子场景的可验证根因表述。

**边界（MUST）**：

- 每个关键数字须回指某次新版 smart-ask-data **第 11 步** `kweaver dataview query` 的原样结果（经 `smart-ask-data`）；禁止编造。
- 拆子问题前须已完成 **子场景识别**，并完成与用户问题相关的新版 smart-ask-data **第 7～9 步**（候选表检索 → 权限校验 → `dataview get` **收敛可查询数据视图与字段**）；且须已阅读本文 **子问题拆解指引 + 输出模板**（见文首「读取时机」）。
- 禁止 `execute_code_sync`、`json2plot`。
- 不得输出须由归因证据链支撑的「根因结论」；证据不足时列 **假设 + 需补查询**（见第 7 节）。
- 子问题数量建议 **2～8** 条（含）。

## 结论验证对齐（MUST）

启用数据解读子场景时，在进入最终输出前必须对“趋势/异常/贡献/建议”结论做可复核校验；任一项不通过，需回到取数编排（调整子问题/SQL/口径）修正后再输出：

- **证据可回指**：主文每个关键数字与判断必须能回指到某条 `dataview query` 的结果行/列；禁止用“感觉/常识”补结论。
- **口径一致**：趋势、异常、贡献引用的指标口径（分母/去重/时间窗/过滤条件）一致；若确有多口径，必须在第 2 节分列说明。
- **算术可验算**：占比/差值/增长率等仅在可验算前提下给出，并写清分母；TopN 截断必须显式说明“仅展示前 N 条”。
- **时间逻辑合理**：出现大量“负时滞/未来日期”等反常时，必须在第 4 节标注为口径冲突/需核查，并避免据此输出强结论。
- **限行不外推**：若任何子问题使用明细 `LIMIT 200`，该结果仅用于样例说明，不得据此推断全量结构或占比；需要全量结构必须改为聚合取数且避免 `LIMIT 200` 截断。

## 输出模板（MUST：启用本子场景时强制按照这个模板来输出）

> 结果表行数较多时可摘要展示，但必须声明是否 TopN 截断。**模式 A（执行型）** 下：每个子问题的 **取数逻辑（非技术语言）+ 结果行数/是否截断** 必须出现，**建议放在第 2 节「附录 / 核对用」或全文末尾**，避免打断业务读者阅读第 1、3～6 节。**对用户最终输出不得展示 SQL**（含附录）；SQL 如需复跑仅做内部留存。
> **标题括号注释禁止输出（MUST）**：每一章节标题后面括号内的内容仅为技术性说明，**对用户可见输出禁止包含括号及其中内容**；仅输出括号前的小标题文本。
- **表格优先（MUST）**：涉及数字的结论尽量用 Markdown 表格承载（如按月趋势表、Top 贡献表、异常点对照表），文字用于解释业务含义；禁止仅用段落叙述“增长/回落/集中”而不给对应数表。

### 1. 解读摘要（1～3 条）

- 每条可回指 **某一子问题** 的结果行 / 列或可验算计算（主文可用「其中一次统计表明…」等说法，**不必**写子问题编号，编号放在第 2 节附录对照表）。
- **语气**：一句话一个结论，先写 **对业务意味着什么**，再带关键数字（如「约占全体的三成」）。

### 2. 口径与证据来源（必须；拆成「业务口径」+「附录」两段）

**业务口径**

- 用 3～8 行说明：**回答的是用户的哪一个问题**、**数据大致覆盖什么范围**（时间、地区、业务对象）、**主要指标在业务上代表什么**、**当前结论受哪些明显限制**（如样本量小、缺字段）。
- **避免**：长表名路径、大段 SQL、接口术语。

### 3. 多维分析（趋势/结构/对比；若无可用维度则「不适用」）

> 目标：不仅回答“随时间怎么变”，还要回答“**在哪些维度上差异最大、贡献最大、组合风险最高**”。本节只写业务化结论；若需标注对应查询，用附录中的子问题编号表，**不在此节逐条复述 SQL**。

- **适用性判定（必须先写一句）**：说明本次是否具备可分析维度（例如：是否有时间字段、地区字段、行业/类型字段、主体属性字段等）。缺任一维度就对该维度写「不适用：缺××字段/缺对应子问题结果」。  
- **度量值口径（必须点名）**：本节所有结论围绕 1～3 个度量（如“企业数/记录数/金额/占比/增长率”等）。每段结论必须带代表数值，并能回指某张结果表中的列。

- **3.1 时间维度（若有时间序列）**  
  - **趋势描述**：写 **时间段 + 变化方向 + 代表数值**（如「2024 至 2025，处罚件数从 36 增至 123」）。  
  - **阶段划分（可选）**：仅当序列支持；用业务语言概括「前半段/后半段」「哪几个月偏高」。  
  - **结构随时间变（可选）**：若结果同时含“类别/地区”等维度，写清“**结构是否迁移**”（例如 Top 类别占比上升/下降）。

- **3.2 空间/地域维度（若有地区/城市/区县等）**  
  - **分布与集中度**：写 TopN 地区及其贡献（件数/占比），并给出“长尾是否明显”。  
  - **对比口径**：若存在分母（如企业总量、人口、存量主体数），优先写 **标准化后指标**（如每万户、占比）；否则明确“未标准化，仅为绝对量对比”。

- **3.3 类别/类型维度（若有违法类型、列入事由、案件类型、行业门类等）**  
  - **主导类别**：写 Top 类别及占比，并说明其与第二梯队差距。  
  - **结构差异**：若同时存在地区/时间，指出“哪些类别在特定地区/时期更突出”。

- **3.4 主体属性维度（若有企业规模、登记状态、成立年限、组织形式等）**  
  - **分层对比**：写“哪一层级更高/更集中”，并给出代表数值（占比/均值/中位数等，取决于结果字段）。  
  - **梯度关系（可选）**：若层级有序（小微→中型→大型），可写“是否存在单调上升/下降”的梯度。

- **3.5 交叉维度（建议至少 1 组；若输入支持）**  
  - **组合热点**：从二维交叉表（如 地区×类型、时间×类型、地区×主体属性）中找“**Top 组合**”与其贡献。  
  - **高风险画像**：用 1～2 句话把“组合特征”概括成可执行的画像（例如“某地区 + 某类型 + 某规模段”），并附代表数值。  

- **3.6 维度一致性与可解释性（可选但推荐）**  
  - **一致性**：说明各维度结论是否相互印证（例如“Top 地区是否也是 Top 类型的主要来源”）。  
  - **解释边界**：若只能描述差异、不能解释原因，明确写「当前仅能描述分布差异，原因需补数/补字段核实」。

### 4. 异常与可能解释（受限）

- **异常点**：主文写 **哪一类 / 哪一时期 / 哪一地区** 明显偏离常态，配数字；技术细节（哪次查询的第几行）放附录或脚注。
- **可能解释**：仅 **证据内解释**（数据本身能直接支撑的说法）或明确标注 **「尚属假设，需再取数核实」**；**不要**用「模型认为」「算法显示」等空泛表述。

### 5. 关键贡献维度（必须尽量给出）

- **Top 贡献项**：主文写 **哪几项排在前面、大约各贡献多少**（件数、金额或占比）；表头尽量用业务列名（如「违法类型」「案件数」），与查询结果字段不一致时须在 2b 说明映射。
- **贡献度**：有占比列则引用；否则可验算占比并写清分母（用业务话写「占全部××的百分之…」）。

### 6. 建议（必须与证据绑定）

- **立刻可做** / **需要补数再定** 两组，共 3～7 条；每条建议开头用 **动作**（如「建议优先核查…」「建议与××部门确认…」），并点明依据来自哪类统计（不必写 SQL）。

### 7. 局限与下一步（必须）

- 写明当前 **还不能下什么结论**（用业务后果表述，如「无法判断是否为全省趋势」）。
- **下一步**：用业务问题列出「还可以多问什么」（如「若需分月走势，请再统计按月汇总」）；若需技术同事落地，可在括号内补一句 **可 SQL 化的大致取数方向**（仍由总入口路由 `smart-ask-data`）。

### 附录
**附录：核对用（必须，可与主文分栏或置于文末「附录：数据与查询依据」）**

- **kn_id**：与各子问题一致（或分列说明）。
- **候选表与字段依据（第 7～9 步）**：第 7 步候选表列表（含 `dataview_id`）、第 8 步判权要点（`operations` 是否含 `data_query`）、第 9 步结构摘要（`meta_table_name` + 关键字段）。
- **子问题清单**：编号；每号 **最小口径** + **取数逻辑（业务话）** + **结果行数/是否截断** + **结果数据索引**（说明哪些表格来自该子问题）。
- **口径复述**：时间窗、过滤、分组、排序（与各 SQL 一致）。

## 模式 A — 执行型（默认）

- **输入**：用户提出“解读/趋势/异常/贡献/建议”等洞察诉求（可附时间范围、过滤条件、指标口径）；用户**未**声明“只消费已有取数结果”。  
- **执行**：按 `smart-data-analysis` → `smart-data-insights` 编排执行，必须先完成新版 `smart-ask-data` **第 7～9 步**（候选表检索 → 判权 → `dataview get` 收敛字段），再阅读本文并拆分子问题，最后完成 **第 10～11 步**（生成 SQL → `dataview query` 执行取数）。  
- **输出**：必须按本文「输出模板（须按此章节顺序）」写满各章；并在附录（2b）中提供可复核证据：
  - **kn_id**、`dataview_id`、`meta_table_name`、关键字段（来自第 7～9 步）
  - **子问题清单**：每条包含 **最小口径 + 取数逻辑（业务话）+ 结果行数/是否截断 + 结果数据（原样）**
  - 若某章节结论依赖某条子问题，需能在附录中定位到对应结果行/列；证据不足则该章节写「不适用/不可判定」并在第 7 节列出**最小补数子问题**。

## 模式 B — 仅消费输入（降级）

- **输入**：用户显式提供 **≥1** 段 `smart-ask-data` 最终交付原文（每段至少含 **kn_id、原样结果、最小口径**；若包含 SQL 仅用于内部核对），且声明 **不再新取数**。
- **输出**：不发起新的第 7～11 步取数（不做候选检索/判权/`dataview get`/生成 SQL/执行查询）；仍须在合成正文前 **阅读本文「子问题拆解指引」与「输出模板」**（读取时机同文首），再核对用户交付是否覆盖模板必选节。若段不足以支撑趋势 / 异常 / 贡献之一，对应章节写不适用并在第 7 节列需补子问题。**对用户主文**仍遵守上文「对用户正文的语言」：业务化叙述 + 附录承载「取数逻辑/证据索引」（不展示 SQL）。

缺失必填项 → **终止解读**，列出补齐清单。
