# 数据洞察 · 场景：对比分析（Contrastive analysis）

> **读取时机（MUST）**：当且仅当本轮 **启用对比分析场景** 时，须在 **子场景识别已完成**、且已完成新版 smart-ask-data **第 7～9 步**（候选表检索 → 权限校验 → `dataview get` **收敛可查询的数据视图与字段**）之后、编写子问题清单之前，阅读本文至少 **「子问题拆分规则」** 与 **「输出模板（对比分析章节）」** 两节。**禁止**未读模板即列出子问题；未启用本场景时 **禁止**整本预读。与其它子场景（含 `attribution_analysis.md`）不必同时预读。

本场景强调：**对比 / 比较 / 同比 / 环比 / 两年 / 两期 / A 与 B** 等对照结构。

## 结论验证对齐（MUST）

启用对比分析子场景时，在进入最终输出前必须对对比结论做可复核校验；任一项不通过，需回到取数编排（调整子问题/SQL/口径）修正后再输出：


- **对比对象清晰且口径一致**：必须能用业务话说清「拿什么和什么比」（本期/对比期或 A/B 的边界），并确保两边的过滤条件、去重口径、时间窗定义一致（除非显式声明差异）。
- **差值/增幅可验算**：\(B-A\)、\((B-A)/A\) 等差值与变化率必须能从结果表验算；分母为 0 的增幅必须按模板写「—」并在主文解释，禁止硬算百分比。
- **分母与单位一致**：同一指标跨两期的单位、币种/金额口径一致；如做“件均金额”等派生指标，需写清分子分母与是否可验算。
- **TopN 截断不伪装全量**：若任何一侧为 TopN/限行结果，必须明确“仅展示前 N 条”，不得据此推断全量差异或占比；需要全量差异时改为聚合取数。

**对用户正文的语言（MUST）**

- **主文**：先说清 **「拿什么和什么比」**（如「2025 年对比 2024 年」），再用 **多了多少 / 增长百分之几 / 哪一类拉开的差距最大** 等口语化结论；少用「组 A、组 B」「B−A」等符号，必要时用「后期相对前期」。
- **变化率**：用「大约是去年的几倍」「比去年多一成」等帮助理解；若分母为 0 无法算增幅，主文写 **「上年为零，本年有数据，不宜写百分比」**。
- **技术核对**：对比表可与附录合并；差值公式 \(B-A\)、\((B-A)/A\) 等 **留给附录** 或对 IT 读者简述即可。
- **表格优先（MUST）**：对比结论（谁高谁低、差多少、增幅多少）必须以“对比表”为主呈现，文字用于解释业务含义；禁止只写一段话不给对照表。

## 触发

用户提到「对比 / 比较 / 同比 / 环比 / 两年 / 两期 / A 与 B」等；或子问题结果需形成 **至少两组** 可比对象。

---

## 子问题拆分规则（MUST）

**必须先识别**：

- **可对比维度**：至少满足一类——**时间**（如 2024 vs 2025、本月 vs 上月）、**类型 / 分组**（如类型 A vs B）、**区域 / 其它枚举二分组**（须在 SQL 或口径中定义清楚 A、B 边界）。
- **对比指标**：须为可聚合度量（与维度分析中的 M 一致）。

**规则**：

- **时间对比**：至少包含「各对比期分别汇总」的子问题（可为一条带 `year` / `period` 分组的 SQL，或两条各查一期的 SQL）；若还要「按类型看两年差异」，则 **每期 × 重点类型** 可再拆，须在子问题清单中列明，避免重复口径。
- **类型 / 其它维度对比**：对每个要对比的切片各一条子问题，或一条 SQL 中显式产出多组可对照列；无论哪种方式，**每条第 11 步查询结果**须在交付物中可区分。
- **合成**：对比表中的数字必须来自明确的一条或多条子问题结果；差值 / 变化率仅在 **可验算** 时写出公式（如 \(B-A\)、\((B-A)/A\)），且注明分母为 0 时不可算。

## 输出模板（对比分析章节; MUST：启用本子场景时强制按照这个模板来输出）

> 适用前提：子问题结果可形成 **至少两组** 可比对象。不满足则写「不适用」，并给出最小增量取数建议。
> **标题括号注释禁止输出（MUST）**：每一章节标题后面括号内的内容仅为技术性说明，**对用户可见输出禁止包含括号及其中内容**；仅输出括号前的小标题文本。

### 比的是什么（必选；对用户用业务话）

- **在比什么**：用完整业务句写清对照双方（如「将 2025 年与 2024 年的处罚案件数与罚没金额对比」），不要用「组 A / 组 B」作为主标签。
- **比的是哪几个指标**：如件数、金额、件均金额等，各用一句话解释 **对业务的含义**。
- **附录**：哪条子问题提供哪一组数、对比期边界与取数逻辑要点（与子问题口径一致）。对用户最终输出（含附录）**不展示 SQL**，如需复跑仅内部留存。

### 对比结果一览（必选）

主文可先给 **一段话总结谁高谁低、差多少**，再给表。表头建议业务化，例如：

| 指标名称 | 第一期（建议写实际年份或期名） | 第二期 | 第二期比第一期多/少 | 增幅或降幅（可选，可验算才写） |

（差值、变化率须在附录或可验算前提下与 SQL 结果一致；分母为 0 时该格写「—」并主文说明原因。）

### 差异主要来自哪里（可选）

若某子问题已提供「时期 × 业务类型」等交叉结果：主文用 **「增量主要由××类型带来」** 等表述，并配简化表；完整交叉表可放附录。

### 对比下的业务结论与建议（必选）

- 3～7 条：面向 **决策或执行**（如「建议下阶段重点跟踪××类型」「建议与××口径对齐后再比 2026」）；避免仅重复表格数字而无行动含义。

### 局限与下一步（必选）

- 主文写 **本次对比不能推出什么**（如「不能代表全国」「2026 年数据未闭合不宜做同比」）。
- **下一步**：业务化写「若要更可信还需要…」；技术取数需求可简短附在附录。

## 与维度分析、数据解读的协作

- **维度分析** 已给出单期分布时：对比章节优先引用 **跨期** 子问题，避免同一度量在同一回复中被两种口径重复计算却不说明。
- **数据解读** 已含同比/环比叙事时：对比章节以 **表格式对照** 为主，叙事可简短；若需指回取数依据，**优先写「与上文数据解读一致」**，子问题编号与取数逻辑要点放 **附录**（不展示 SQL），避免主文重复技术标签。
