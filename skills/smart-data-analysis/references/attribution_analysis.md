# 数据洞察 · 场景：归因分析（Attribution analysis）

> **读取时机（MUST）**：当且仅当本轮 **启用归因分析子场景**（`attribution_analysis`）时，须在 **子场景识别已完成**、且已完成新版 smart-ask-data **第 7～9 步**（候选表检索 → 权限校验 → `dataview get` **收敛可查询的数据视图与字段**）之后、编写子问题清单之前，阅读本文至少 **「子问题拆解指引（MECE）」** 与 **「输出模板（归因分析章节）」** 两节。**禁止**未读模板即列出 MECE 子问题；未启用本场景时 **禁止**整本预读。

本场景的定位：**在可复核的多子问题取数证据链上**，输出「根因候选 / 贡献拆分 / 影响范围 / 建议草案」等归因型结论；**完整标准报告正文**（与 `smart-reporting` 场景 `attribution_analysis_report` 同构的排版）仍由总入口在同轮 **证据包就绪后** 交给 `smart-reporting` 组装。与 **数据解读**、**维度分析**、**对比分析** 共享同一套上游编排（**子场景识别** → 第 7～9 步 → **读本场景模板** → MECE 子问题 → 第 10～11 步 `dataview query`），差异在于 **子问题数量更多、强调 MECE 与可验证根因**、以及 **可选二次加工**（见下文「能力边界」）。

**对用户正文的语言（MUST）**

- **主文**：先说清「**在比什么、异常是什么、证据指向哪几条子问题**」，再用业务话写根因候选与建议；避免无编号地堆砌 SQL。
- **根因表述**：每条根因必须写 **子问题编号 + 结果中的数字/分组**；禁止无证据的「肯定是××原因」。
- **技术核对**：原样 SQL、大段结果表、候选表列表放在 **附录** 或「口径与证据来源」后半。

## 结论验证对齐（MUST）

启用归因分析子场景时，在进入最终输出前必须对“根因候选/贡献拆分/影响范围/建议”做可复核校验；任一项不通过，需回到取数编排（调整 MECE 子问题/SQL/口径）修正后再输出：

- **证据链闭环**：每条根因必须能回指到某个子问题的 `dataview query` 结果（行/列/分组数字），禁止“只有方向没有证据”的根因陈述。
- **MECE 不重不漏**：子问题之间不重复计数、不遗漏关键切片；若发现重复/遗漏，必须重拆子问题而不是用文字补丁掩盖。
- **算术可验算**：贡献拆分、差值分摊、占比等必须能从结果表验算（或在使用 `execute_code_sync` 时能回指输入输出并解释验算方式）。
- **口径一致性**：本期/对比期、异常起点、粒度、分母/去重口径在子问题间一致；多口径并存必须在口径卡片里显式声明。
- **稳健性（轻量交叉验证）**：对最关键的 1～2 条根因做替代口径/交叉维度验证（例如换粒度、换分组维度或做总量校验），冲突则回滚重查并在局限中说明。

## 触发与边界

**触发**：用户提到「归因 / 根因 / 为什么 / 驱动因素 / 下滑原因 / 下跌原因 / 转化变差 / 流失原因 / MECE / 证据链 / 深度归因」等，且需要 **多子问题取数** 支撑结论。

**边界（MUST）**

- **取数路径**：每个子问题的指标数字须来自新版 smart-ask-data **第 11 步** `kweaver dataview query` 的**原样结果**；`FROM`/`JOIN` 表名须来自第 9 步 `meta_table_name`，字段须在 `dataview get` 的 `fields` 内。
- **编排顺序**：拆 MECE 子问题前须完成 **子场景识别** + smart-ask-data的第 7～9 步 + 本文 **子问题拆解指引 + 输出模板** 阅读（见文首「读取时机」）。
- **子问题数量**：建议 **3～12** 条（含）；超过时应先收敛口径或分轮；须遵守「停止规则」避免无限下钻。
- **可选工具（仅本子场景）**：当 SQL 难以表达贡献占比、差值分摊、多结果合并等时，允许在证据链清晰的前提下使用 **`execute_code_sync`**；**输入必须**来自本子场景已取得的 `dataview query`（及链路上已声明的中间结果），**禁止**无源加工。细节命令与安全边界以 [`execute-code-sync.md`](execute-code-sync.md)为准。
- **与 `smart-reporting` 分工**：本子场景交付 **证据包 + 归因要点草案**；**不**在同 skill 内替代 `smart-reporting` 写满「归因分析报告」全文章节（除非用户明确只要证据包、不要报告）。

**与「数据解读」子场景的关系**

- 若同轮 **同时启用** `data_interpretation` 与 `attribution_analysis`：**轻量趋势/异常叙述**放在数据解读章节；**可验证根因、贡献分解、MECE 证据链**放在归因分析章节。数据解读 **不得**抢先给出无子问题支撑的「最终根因」。

## 子问题拆解指引（MECE，与输出章节对齐）

| 输出章节 / 证据块 | 典型子问题方向（须按第 9 步 `fields` 改写为可执行 SQL） |
| --- | --- |
| **趋势 / 异常起点** | 按日/周/月的核心指标序列；标注异常开始时段 |
| **幅度与对比** | 本期 vs 对比期总量与增速（口径须在子问题标题与 SQL 中显式） |
| **维度贡献** | 区域/品类/渠道/用户分群等 Top 贡献或降幅贡献（各一条或一条多指标） |
| **漏斗 / 链路**（若字段支持） | 曝光→点击→下单→支付等分步转化 |
| **供给侧 / 活动**（若字段支持） | 库存、履约、活动标签等与指标交叉 |

**MECE 要求**：子问题之间 **不重不漏**；每个子问题 **一条主 SQL**（允许另条仅取对比期对称查询）；清单须在证据包中可逐条对应 SQL+结果。

**停止规则（满足其一可收敛）**

- Top 维度对总差额的解释度达到业务约定阈值（须在口径卡片写明，未给则写「待业务确认」）；
- 漏斗或时间序列上出现 **可定位的单点断裂** 且有多子问题一致支撑；
- 继续下钻将超出字段可支撑范围 → 在局限中写明并停止新增子问题。

---

## 失败门禁（本场景）

- 第 7～9 步任一失败 → 终止，禁止盲写归因 SQL。
- 任一子问题第 11 步失败且无法补救 → 该子问题标为失败；依赖其的「根因结论」写 **不适用 / 不可判定**，禁止用其它子问题数字顶替。
- **禁止**：无子问题结果的「根因定论」；禁止与口径卡片矛盾的 SQL 仍声称支撑根因。

## 输出模板（归因分析章节；MUST：启用本子场景时强制按照这个模板来输出）

> 若用户同轮还需要 `smart-reporting` 的 **`attribution_analysis_report`**：下列 **证据包必备块** 须完整产出，便于原样粘贴为报告输入；**主文业务叙述** 可与报告章节合并编排，但 **不得省略证据块**。
> **标题括号注释禁止输出（MUST）**：每一章节标题后面括号内的内容仅为技术性说明，**对用户可见输出禁止包含括号及其中内容**；仅输出括号前的小标题文本。
- **表格优先（MUST）**：归因链路中的关键证据必须尽量以表格呈现（MECE 子问题清单表、各子问题结果表、贡献拆分表、影响范围 TopN 表）。文字用于解释“证据指向什么/意味着什么/建议做什么”，禁止只写结论而不给对应证据表。

### 1. 归因摘要（1～5 条）

每条：**业务结论 + 子问题编号引用**（例：「Q3 表明华东贡献下降占整体降幅约六成——为读数结论，非外部归因」）。

### 2. 口径卡片（必须）

与 Step 2 等价物：分析类型、主体、指标定义、本期+对比期、粒度、异常阈值（缺失须声明默认或待确认）。

### 3. 分析路径与子问题清单（MECE，必须）

表格：**编号 / 子问题目标 / 依赖维度或表（若已知）/ 预期输出指标 / 与口径卡片对齐说明**。

### 4. 数据与证据（必须；按子问题逐个）

每个子问题固定顺序：

1. **取数逻辑（业务话）**：说明“从哪张数据视图/哪类记录取数、怎么过滤、按什么分组、算什么指标、如何排序、是否 TopN/限行”（对用户最终输出禁止展示 SQL）。
2. **结果数据（原样）**：正文优先 Markdown **表格**
3. （可选）**二次加工说明**：若使用 `execute_code_sync`，写清输入来自哪条子问题结果、输出了什么、如何回指验算

### 5. 归因结论（1～5 条可验证根因）

每条：**根因陈述 + 子问题编号 + 结果行/列要点**；禁止超出证据链外推。

### 6. 影响范围（基于数据）

仅当子问题结果已给出城市/品类/渠道等维度时撰写。

### 7. 建议（立即 / 中期 / 长期）

每条建议须对应第 5 节某条根因（写清「对应根因 #x」）。

### 8. 局限与下一步（必须）

写清 **尚不能推出的结论**、需补字段/补表/补时间窗的子问题建议（可 SQL 化）。

### 9. 证据包与报告交接（若用户要归因报告）

在同轮由总入口将本节 **证据包原文**（含口径卡片、MECE 清单、每子问题 SQL+结果、可选加工说明、本节归因要点）交给 [`smart-reporting.md`](smart-reporting.md) 场景 **`attribution_analysis_report`**；**同一 assistant 回复内** 紧随报告章节，**禁止**要求用户下一条再发 `/smart-reporting`。

## 模式 A — 执行型（默认）

- **输入**：用户提出「归因/根因/为什么/驱动因素/MECE/证据链」等诉求，且**允许重新取数**（未声明“只消费已有归因证据包”）。  
- **执行**：按 `smart-data-analysis` → `smart-data-insights`（`attribution_analysis` 子场景）编排执行；必须先完成新版 `smart-ask-data` **第 7～9 步**（候选表检索 → 判权 → `dataview get` 收敛字段），再阅读本文并拆分 MECE 子问题，最后完成 **第 10～11 步**（生成 SQL → `dataview query` 执行取数）。必要时可使用 `execute_code_sync` 做二次加工，但其输入必须来自已取数的子问题结果（见上文边界）。  
- **输出**：必须严格按本文「输出模板（归因分析章节）」输出第 1～8 节（以及在需要时输出第 9 节交接说明），并形成**可复核证据包**：  
  - **口径卡片**完整（指标定义、本期/对比期、粒度、阈值/异常起点等）  
  - **MECE 子问题清单**完整且可逐条对应  
  - 每个子问题均给出 **取数逻辑（业务话）+ 结果数据（原样）**（正文优先表格；对用户最终输出（含附录）禁止展示 SQL；SQL 如需复跑仅内部留存）  
  - 每条根因结论必须写清 **子问题编号 + 对应结果行/列数字**；证据不足则写「不可判定」并在第 8 节列最小补数清单  

## 模式 B — 仅消费输入（降级）

- **输入**：用户显式提供 **完整归因证据包**（含口径卡片、MECE 清单、每子问题结果与最小口径；若包含 SQL 仅用于内部核对），且声明 **不再发起新的第 7～11 步取数**。
- **输出**：不执行新的 `kn-schema-search` / 判权 / `dataview get` / `dataview query`；仍须在合成前阅读本文 **子问题拆解指引 + 输出模板**，再按第 1～8 节重组或续写归因叙事；缺块则在第 8 节列补齐清单。对用户最终输出不展示 SQL，改用「取数逻辑 + 证据索引」。
